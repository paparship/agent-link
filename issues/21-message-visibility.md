# 21 — 消息/任务可见性：状态面板、已读回执、数据保留

Type: Design

## 问题

当前消息和任务的通信模型假设"双方在线、即时处理"，但 agent 的实际工作流完全不同：

1. **agent 执行期间完全失明** — 一个长任务跑半小时，期间所有消息和任务都无法感知
2. **发送者盲区** — 发出消息或任务后，不知道对方是否收到、是否在忙、是否离线
3. **消息不可追溯** — 发送后无法查询"对方读了没有"
4. **无限堆积** — inbox 和 task 记录没有 TTL，死信积累

## 设计目标

1. **发送者知情** — 每次 send / task send 的响应带回接收方实时状态面板
2. **消息可查** — 消息有 ID、有状态（已投递/已读取）、可查询
3. **统一面板** — 消息和任务用同一套状态描述，接收方是忙/闲/离线一目了然
4. **数据可清理** — 所有数据有 TTL 或配额，不无限堆积

## 状态面板格式

send / task send 的响应（成功或失败）统一附加以下上下文：

```
✓ 已投递 / ✗ 无法投递

接收方: supermicro main session
  状态:   忙碌 / 空闲 / 离线
  当前:   task #deploy-042（已进行 14 分钟）
  积压:   3 条待读取消息
  最后在线: （离线时显示）2 小时前
```

成功时：
- 消息 id 显示在面板后面
- 忙碌时提示"将在当前任务完成后被读取"

失败时（task 忙拦截）：
- 除状态面板外提示"当前任务完成后即可接收新任务"
- 引导用 --interrupt

## 状态转换

### 消息生命周期

```
sent ──→ delivered（pull 拉取）──→ read（可选，拆包后）
  │                                       │
  └──→ expired（TTL 到期）               └──→ deleted（TTL）
```

### 状态存储变化

当前：`LPUSH agentlink:inbox:<device>:<session>` + `RPop`（消费即删）

改为：
- inbox 仍用 `LPUSH` 作为队列
- 同时写一条独立消息记录：`agentlink:msg:<msg_id>`（带 TTL）
- `RPop` 时标记 `delivered_at`
- 保留 TTL 内可查
