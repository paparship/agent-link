# 17 — GitHub Actions CI：自动测试 + 构建 + 发布

Type: AFK

## 背景

当前开发和发布流程全部手动操作：`git push` → 本地 `go test` → `make build-all` → 打开 GitHub 拖文件上传 Release。没有自动质量门禁，发布步骤繁琐易错。

需要一个 CI 流程，在 push 时自动测试和编译，在打 tag 时自动发布 Release。

## 设计方案

### 工作流总览

单一文件 `.github/workflows/ci.yml`，三个 job 按触发条件区分：

```
push / PR  ─→  test job     ← golangci-lint + go test
    │
    └────────→  build job    ← make build + make build-server（依赖 test 通过）
    
push tag v* ─→  release job  ← make build-all → GitHub Release（依赖 test + build 通过）
```

### 1. test job

**触发：** 所有 push + PR

```yaml
steps:
  - uses: actions/setup-go@v5 (go 1.24)
  - make lint        # go vet ./...
  - make test        # go test ./... -count=1
```

### 2. build job

**触发：** 所有 push（不含 tag），等待 test 通过

```yaml
needs: test
steps:
  - make build       # 编译 agentlink（当前平台）
  - make build-server # 编译 server
```

### 3. release job

**触发：** push tag `v*`，等待 test + build 通过

```yaml
needs: [test, build]
steps:
  - make build-all   # 4 平台交叉编译
  - 用 softprops/action-gh-release 创建 Release
  - 上传 bin/agentlink-* + install.sh
```

### 使用 GitHub 内置能力

- **GITHUB_TOKEN**：Release 创建无需额外 secrets
- **actions/setup-go**：Go 环境无需额外配置
- **softprops/action-gh-release**：社区 action，自动上传 assets

### 设计要点

- 单一文件比拆分 3 个文件更简洁，job 间通过 `needs` 串联
- `release` job 只在 tag 时触发，不影响日常 push
- `install.sh` 作为 release asset 上传，确保用户下载的始终是对应版本的安装脚本
- `build-all` 产物在 `bin/` 目录，与 `.gitignore` 一致

## 涉及文件

| 文件 | 改动 |
|------|------|
| `.github/workflows/ci.yml` | **新增** — CI 配置文件 |

## 不做的

- 不自动部署 server（需要手动确认）— 后续可加 deploy job
- 不接入 codecov / sonarqube — 项目早期不需要
- 不自动打 tag — 发布时机由你决定
- 不跨平台编译 server（只在 linux/amd64 运行）
- 不构建 Docker 镜像

## Acceptance criteria

- [ ] push 到任意分支时自动运行 `make lint` + `make test`
- [ ] push 到任意分支时自动运行 `make build` + `make build-server`
- [ ] build job 在 test job 失败时自动跳过
- [ ] push tag `v*` 时自动运行 `make build-all`
- [ ] push tag `v*` 时自动创建 GitHub Release 并上传 4 个 binary + install.sh
- [ ] Release 标题和正文自动使用 tag 名称
- [ ] 非 tag push 不会触发 Release
